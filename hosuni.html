# hosuni — tool-use daemon for local inference

A compiled Rust daemon that gives local models hands. Receives
input from any interface, routes inference through squared, parses
completions for tool calls, executes them, feeds results back.
One binary, any node.

Named for the 1988 Seoul Olympics mascot pair. Hodori is the
governor. Hosuni is the daemon that acts on his behalf.

## What it is

A persistent process that:

  1. Sleeps on file descriptors (Discord WebSocket, local port)
  2. Wakes when input arrives
  3. Checks lookdown (personal first, universal fallback)
  4. If no match, routes to inference through squared
  5. Parses the completion for tool calls
  6. Executes the tool (read file, run command, osascript, etc.)
  7. Appends the result to the conversation
  8. Sends the next turn to inference
  9. Returns the final response through the interface it came from
  10. Logs the session, goes back to sleep

## What it is not

  - Not a chatbot. A daemon that executes.
  - Not model-specific. Talks to squared's socket; whatever model
    is behind it is interchangeable.
  - Not platform-specific. Compiles to a static binary. Runs on
    Mac, Linux, Pi.

## Architecture

  shell (subtract.sh, handler.sh)     user-facing, orchestration
  rust daemon (hosuni)                 inference routing, tool execution, state
  c proxy (squared)                    network, multiplexing, TLS
  llama-server                         inference

Each layer is a compiled binary or a shell script. No interpreters
on the deploy path.

## Interfaces (ears)

  Discord WebSocket     DM and channel messages, whitelist-gated
  Local port            for terminal, scripts, cron, other daemons

Each interface is a listener that feeds input to the same core.
Adding an interface does not change the core. Removing one does
not break the others.

## Core loop

  input arrives on any interface
    → check lookdown.personal.tsv (T0, instant)
    → check lookdown.tsv (T0, instant)
    → if match: return, log, done
    → build messages array (conversation state for this channel)
    → POST to squared endpoint (local inference)
    → read completion
    → if completion contains tool call:
        → execute tool
        → append result to messages
        → POST again (multi-turn)
        → repeat until no tool call
    → if local returns empty: fallback to claude -p -c
    → return response through originating interface
    → log session

## Tool definitions

A file the daemon reads at startup. Not compiled in.

  ~/.hosuni/tools.tsv

  Format: name<TAB>description<TAB>command_template

  read_file     Read a file             cat {path}
  write_file    Write to a file         tee {path}
  list_dir      List directory          ls -la {path}
  run_command   Run a shell command     sh -c {command}
  osascript     Run AppleScript         osascript -e {script}

osascript is present on Mac nodes, absent on Linux/Pi. The daemon
does not error on missing tools — it simply does not offer them
to the model.

The tool definition file is the capability boundary. Tools not
in the file do not exist for the model. This is subtractive
security: the daemon can do what the file says, nothing else.

## Conversation state

Per-channel, held in memory during the conversation. Written to
disk as subtract.sh-compatible TSV:

  ~/.subtract/log/hosuni-{channel_id}.log

  Format: timestamp<TAB>Q|A<TAB>content

Conversation history is sent to the model as a messages array.
Truncated to last 20 turns to stay within context limits.

## Inference routing

  1. Read inference endpoint from ~/.subtract/inference_host
     and ~/.subtract/inference_port
  2. POST to http://{host}:{port}/v1/chat/completions
     (OpenAI-compatible, through squared or direct)
  3. If local returns empty or errors: fall back to
     claude -p -c (cloud, continues conversation)

The routing follows subtract.sh's existing model. The daemon
does not choose models. It talks to whatever is behind the port.

## Session logging

Every interaction logged in two places:

  1. Session TSV (per-channel, subtract.sh format)
  2. Daemon log (~/human/sessions/hosuni-discord.log format,
     includes inference source: local|cloud)

## Build and deploy

  Build:    cargo build --release (on rousseau)
  Output:   target/release/hosuni (single static binary)
  Deploy:   scp hosuni {node}:~/.subtract/bin/hosuni
  Run:      ~/.subtract/bin/hosuni
  Service:  launchd plist (Mac) or systemd unit (Linux)

Build-time dependencies (cargo crates, on rousseau only):
  tokio       async runtime (sleep on file descriptors)
  serde       JSON parsing (inference API, Discord Gateway)
  serde_json  JSON serialization
  tungstenite Discord WebSocket (or tokio-tungstenite)

Runtime dependencies: none. The binary and the kernel.

## Configuration

  ~/.keymaster/jean-discord-token.txt     Discord bot token
  ~/.hosuni-discord-whitelist             allowed Discord user IDs
  ~/.hosuni/tools.tsv                     tool definitions
  ~/.subtract/inference_host              inference endpoint host
  ~/.subtract/inference_port              inference endpoint port
  ~/.subtract/lookdown.tsv                universal lookdown
  ~/.subtract/lookdown.personal.tsv       personal lookdown

All plain text files. No database. No config format beyond TSV
and single-value files.

## What the daemon does not do

  - Does not choose which model to load (that is llama-server's job)
  - Does not manage model files (that is the operator's job)
  - Does not proxy HTTP (that is squared's job)
  - Does not handle shell integration (that is subtract.sh's job)
  - Does not sign artifacts (that is ssh-keygen's job)
  - Does not manage users (that is the kernel's job)

## Testing

The variance-lab harness runs through hosuni. The five routing
table prompts (code-refactor, identity-planning, jean-heartbeat,
research-tooling-landscape, security-audit) dispatched through
the daemon with tools enabled. This is the first test of local
models with hands.

Finding 4 from the routing table: "single-turn harness is not
valid for reasoning tier grading." The daemon IS the multi-turn
harness that makes the grading real.
